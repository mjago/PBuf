1 # Theory of operation
2 
3 This is a description of the theory of operation of *PBuf*.
4 
5 ## Configuration
6 
7 *PBuf* is configured at compile time by setting a small number of defines in priority_buffer.h:
8 
9 ```
10 
11 #define BUFFER_SIZE 4 /* may be between 4 to 256 elements in size */
12 
13 #define PRIORITY_SIZE 3 /* may be between 2 to 8 priorities in size */
14 
15 #define ELEMENT_SIZE 8 /* may be either 8, 16, 32, or 64 bits */
16 
17 #define EXTERNAL_DATA_BUFFER /* Enable headless mode if defined */
18 
19 ```
20 
21 The compiler checks these settings at compile time and compile will fail if they are out of limits.
22 
23 These settings may also be provided on the command line to the compiler such as `-DELEMENT_SIZE=8`
24 
25 priority_buffer.h is purposefully kept very minimal to highlight these user configurations.
26 
27 ## Design choice
28 
29 The design is optimised for smaller embedded devices, and should be suitable down to 8-bit devices.
30 
31 ## Requirement Assumptions
32 
33 Due to the nature and size of the project it was decided to push through a design and then refactor
34 for any wrong assumptions in the design and implementation. The design is unit-tested to ensure this
35 refactoring is as painless as possible.
36 
37 ## Memory
38 
39 The circular buffer is a structure containing elements upto the defined `BUFFER_SIZE`. Each element is
40 a structure containing both data and a forward-linking link to the following buffer location.
41 
42 We also have a tail pointer and and array of head pointers whose size is equal to the `PRIORITY_SIZE`.
43 In addition, a further byte is used to store active flags for each priority.
44 
45 With these pointers into the structure we can process both insert and retrieve operations very quickly.
46 The only time we spend following the links to any extent (beyond one or two) is when we are requred to
47 calculate the size of the buffer contents since it was decided the memory saved here was justifiable
48 (the print function also follows the links, but this is a `DEBUG` enabled function only - for the
49 purpose of the command line evaluation program).
50 
51 ## Headless Operation
52 
53 *PBuf* can be used in a headless mode where the user supplies the buffer, and configures PBUF appropriately.
54 In this configuration the commands `PBUF_insertIndex()` and `PBUF_retrieveIndex()` are used to have *PBuf*
55 figure out the appropriate indices to reference the remote buffer correctly. This mode is achieved by
56 defining `EXTERNAL_DATA_BUFFER`, which causes the compiler to build *PBuf* without the internal buffer storage
57 or the buffer manipulation instruction `PBUF_insert()` and `PBUF_retrieve()`.

59 ## Pointers
60 
61 As mentioned, a number of pointers are used to provide fast access into the buffer:
62 
63 Here P2 refers to highest priority, P0 to lowest priority etc.
64 Tail is the tail pointer.
65 Head is the head pointer.
66 The active array maintains the active status.
67 the x denotes an empty element.
68 
69 ```
70 
71 ---------------------<------------------
72 | |
73 -> P2 -> P1 -> P0 -> x >-
74  ^ ^ ^ ^
75  Head[2] Head[1] Head[0] Tail
76 
77 Active[1] = ACTIVE
78 Active[2] = ACTIVE
79 Active[3] = ACTIVE

81 ```

83 ## Retrieving Data
84 
85 To retrieve an element the tail updates by following the link of the element it refernces, and the data at
86 its new position is retrieved. If the head pointer matching the retrieved element's priority has the same
87 index value as the tail pointer, the relevant active flag is set to `NON_ACTIVE`.
88 
89 If there is no data in the buffer all active flags are `NON_ACTIVE` and a suitable message returned to the
90 caller. The empty buffer check is a single instruction operation.
91 
92 ## Adding Data
93 
94 Since we prioritise the data added to the buffer the direction of priorities can be considered as flowing away
95 from the tail from oldest high priority through newest high priority all the way down to oldest low priority
96 through to newest low priority. This means that data is always retrieved highest priority first but in a first-
97 in-first-out manner considering the priority.
98 
99 Elements are added to the buffer by the following means:
100 
101 ### Buffer Empty
102 
103 If the buffer is empty, we add data to the location linked by the tail pointer. We ensure the data's
104 priority is noted on the relevant active flag, and assign the relevant priority head pointer with the
105 array index of the element's position in the buffer. In this way the priority head pointers always point
106 to the newest element having that priority. If any active flag is `INACTIVE` the associated head pointer's
107 data is contains a meaningless value.
108 
109 ### Buffer containing data but not full
110 
111 If the buffer is neither empty nor full, we add data to the location with the least highest priority.
112 This is simply the index value of the lowest active priority pointer. In this case the data may be
113 of a higher priority than its predecesser in which case we need to move it to the correct position. This is
114 dealt with below by remapping the buffer.
115 
116 ### Buffer Full
117 
118 If the buffer is full we need to determine if there is older data we can overwrite rather than throwing away
119 this new data. This is achieved by the following strategy:
120 
121 If there is data in the buffer having a priority that is either equal or of a lesser priority then we pick the
122 lowest priority, and the oldest entry within that priority. We can reference this value by examining the priority
123 head pointers and picking the next highest active priority to the one we want and following its link. This is where
124 we need to overwrite our new data. It may happen that there is no higher priority active in which case our location
125 is linked to by the value of the tail pointer index. Once we've overwritten the older data we will need to rearrange
126 our links to relocate the latest entry to be at the end of it's queue of identical priority. This is described below.
127 
128 ## Remapping the buffer
129 
130 When data of varying priorities are added to the buffer we then need to rearrange the buffer to place the new element
131 at the correct position in the buffer relevant to its priority. We are only concerned with remapping a single element
132 since this is what we added, and the operation is very simple and quick.
133 
134 NOTE: This process is described more thoroughly in the `adding_data_to_the_buffer` document.
