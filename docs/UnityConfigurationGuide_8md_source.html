1 # Unity Configuration Guide
2 
3 ## C Standards, Compilers and Microcontrollers
4 
5 The embedded software world contains its challenges. Compilers support different
6 revisions of the C Standard. They ignore requirements in places, sometimes to
7 make the language more usable in some special regard. Sometimes it's to simplify
8 their support. Sometimes it's due to specific quirks of the microcontroller they
9 are targeting. Simulators add another dimension to this menagerie.
10 
11 Unity is designed to run on almost anything that is targeted by a C compiler. It
12 would be awesome if this could be done with zero configuration. While there are
13 some targets that come close to this dream, it is sadly not universal. It is
14 likely that you are going to need at least a couple of the configuration options
15 described in this document.
16 
17 All of Unity's configuration options are `#defines`. Most of these are simple
18 definitions. A couple are macros with arguments. They live inside the
19 unity_internals.h header file. We don't necessarily recommend opening that file
20 unless you really need to. That file is proof that a cross-platform library is
21 challenging to build. From a more positive perspective, it is also proof that a
22 great deal of complexity can be centralized primarily to one place to
23 provide a more consistent and simple experience elsewhere.
24 
25 
26 ### Using These Options
27 
28 It doesn't matter if you're using a target-specific compiler and a simulator or
29 a native compiler. In either case, you've got a couple choices for configuring
30 these options:
31 
32 1. Because these options are specified via C defines, you can pass most of these
33 options to your compiler through command line compiler flags. Even if you're
34 using an embedded target that forces you to use their overbearing IDE for all
35 configuration, there will be a place somewhere in your project to configure
36 defines for your compiler.
37 2. You can create a custom `unity_config.h` configuration file (present in your
38 toolchain's search paths). In this file, you will list definitions and macros
39 specific to your target. All you must do is define `UNITY_INCLUDE_CONFIG_H` and
40 Unity will rely on `unity_config.h` for any further definitions it may need.

43 ## The Options

45 ### Integer Types
46 
47 If you've been a C developer for long, you probably already know that C's
48 concept of an integer varies from target to target. The C Standard has rules
49 about the `int` matching the register size of the target microprocessor. It has
50 rules about the `int` and how its size relates to other integer types. An `int`
51 on one target might be 16 bits while on another target it might be 64. There are
52 more specific types in compilers compliant with C99 or later, but that's
53 certainly not every compiler you are likely to encounter. Therefore, Unity has a
54 number of features for helping to adjust itself to match your required integer
55 sizes. It starts off by trying to do it automatically.
56 
57 
58 ##### `UNITY_EXCLUDE_STDINT_H`
59 
60 The first thing that Unity does to guess your types is check `stdint.h`.
61 This file includes defines like `UINT_MAX` that Unity can use to
62 learn a lot about your system. It's possible you don't want it to do this
63 (um. why not?) or (more likely) it's possible that your system doesn't
64 support `stdint.h`. If that's the case, you're going to want to define this.
65 That way, Unity will know to skip the inclusion of this file and you won't
66 be left with a compiler error.
67 
68 _Example:_
69 ```C
70 #define UNITY_EXCLUDE_STDINT_H
71 ```
72 
73 
74 ##### `UNITY_EXCLUDE_LIMITS_H`
75 
76 The second attempt to guess your types is to check `limits.h`. Some compilers
77 that don't support `stdint.h` could include `limits.h` instead. If you don't
78 want Unity to check this file either, define this to make it skip the inclusion.
79 
80 _Example:_
81 ```C
82 #define UNITY_EXCLUDE_LIMITS_H
83 ```
84 
85 If you've disabled both of the automatic options above, you're going to have to
86 do the configuration yourself. Don't worry. Even this isn't too bad... there are
87 just a handful of defines that you are going to specify if you don't like the
88 defaults.
89 
90 
91 ##### `UNITY_INT_WIDTH`
92 
93 Define this to be the number of bits an `int` takes up on your system. The
94 default, if not autodetected, is 32 bits.
95 
96 _Example:_
97 ```C
98 #define UNITY_INT_WIDTH 16
99 ```
100 
101 
102 ##### `UNITY_LONG_WIDTH`
103 
104 Define this to be the number of bits a `long` takes up on your system. The
105 default, if not autodetected, is 32 bits. This is used to figure out what kind
106 of 64-bit support your system can handle. Does it need to specify a `long` or a
107 `long long` to get a 64-bit value. On 16-bit systems, this option is going to be
108 ignored.
109 
110 _Example:_
111 ```C
112 #define UNITY_LONG_WIDTH 16
113 ```
114 
115 
116 ##### `UNITY_POINTER_WIDTH`
117 
118 Define this to be the number of bits a pointer takes up on your system. The
119 default, if not autodetected, is 32-bits. If you're getting ugly compiler
120 warnings about casting from pointers, this is the one to look at.
121 
122 _Example:_
123 ```C
124 #define UNITY_POINTER_WIDTH 64
125 ```
126 
127 
128 ##### `UNITY_SUPPORT_64`
129 
130 Unity will automatically include 64-bit support if it auto-detects it, or if
131 your `int`, `long`, or pointer widths are greater than 32-bits. Define this to
132 enable 64-bit support if none of the other options already did it for you. There
133 can be a significant size and speed impact to enabling 64-bit support on small
134 targets, so don't define it if you don't need it.
135 
136 _Example:_
137 ```C
138 #define UNITY_SUPPORT_64
139 ```

142 ### Floating Point Types
143 
144 In the embedded world, it's not uncommon for targets to have no support for
145 floating point operations at all or to have support that is limited to only
146 single precision. We are able to guess integer sizes on the fly because integers
147 are always available in at least one size. Floating point, on the other hand, is
148 sometimes not available at all. Trying to include `float.h` on these platforms
149 would result in an error. This leaves manual configuration as the only option.
150 
151 
152 ##### `UNITY_INCLUDE_FLOAT`
153 
154 ##### `UNITY_EXCLUDE_FLOAT`
155 
156 ##### `UNITY_INCLUDE_DOUBLE`
157 
158 ##### `UNITY_EXCLUDE_DOUBLE`
159 
160 By default, Unity guesses that you will want single precision floating point
161 support, but not double precision. It's easy to change either of these using the
162 include and exclude options here. You may include neither, either, or both, as
163 suits your needs. For features that are enabled, the following floating point
164 options also become available.
165 
166 _Example:_
167 ```C
168 //what manner of strange processor is this?
169 #define UNITY_EXCLUDE_FLOAT
170 #define UNITY_INCLUDE_DOUBLE
171 ```
172 
173 
174 ##### `UNITY_EXCLUDE_FLOAT_PRINT`
175 
176 Unity aims for as small of a footprint as possible and avoids most standard
177 library calls (some embedded platforms don’t have a standard library!). Because
178 of this, its routines for printing integer values are minimalist and hand-coded.
179 Therefore, the display of floating point values during a failure are optional.
180 By default, Unity will print the actual results of floating point assertion
181 failure (e.g. ”Expected 4.56 Was 4.68”). To not include this extra support, you
182 can use this define to instead respond to a failed assertion with a message like
183 ”Values Not Within Delta”. If you would like verbose failure messages for floating
184 point assertions, use these options to give more explicit failure messages.
185 
186 _Example:_
187 ```C
188 #define UNITY_EXCLUDE_FLOAT_PRINT
189 ```
190 
191 
192 ##### `UNITY_FLOAT_TYPE`
193 
194 If enabled, Unity assumes you want your `FLOAT` asserts to compare standard C
195 floats. If your compiler supports a specialty floating point type, you can
196 always override this behavior by using this definition.
197 
198 _Example:_
199 ```C
200 #define UNITY_FLOAT_TYPE float16_t
201 ```
202 
203 
204 ##### `UNITY_DOUBLE_TYPE`
205 
206 If enabled, Unity assumes you want your `DOUBLE` asserts to compare standard C
207 doubles. If you would like to change this, you can specify something else by
208 using this option. For example, defining `UNITY_DOUBLE_TYPE` to `long double`
209 could enable gargantuan floating point types on your 64-bit processor instead of
210 the standard `double`.
211 
212 _Example:_
213 ```C
214 #define UNITY_DOUBLE_TYPE long double
215 ```
216 
217 
218 ##### `UNITY_FLOAT_PRECISION`
219 
220 ##### `UNITY_DOUBLE_PRECISION`
221 
222 If you look up `UNITY_ASSERT_EQUAL_FLOAT` and `UNITY_ASSERT_EQUAL_DOUBLE` as
223 documented in the big daddy Unity Assertion Guide, you will learn that they are
224 not really asserting that two values are equal but rather that two values are
225 "close enough" to equal. "Close enough" is controlled by these precision
226 configuration options. If you are working with 32-bit floats and/or 64-bit
227 doubles (the normal on most processors), you should have no need to change these
228 options. They are both set to give you approximately 1 significant bit in either
229 direction. The float precision is 0.00001 while the double is 10-12.
230 For further details on how this works, see the appendix of the Unity Assertion
231 Guide.
232 
233 _Example:_
234 ```C
235 #define UNITY_FLOAT_PRECISION 0.001f
236 ```

241 ##### `UNITY_EXCLUDE_STDDEF_H`
242 
243 Unity uses the `NULL` macro, which defines the value of a null pointer constant,
244 defined in `stddef.h` by default. If you want to provide
245 your own macro for this, you should exclude the `stddef.h` header file by adding this
246 define to your configuration.
247 
248 _Example:_
249 ```C
250 #define UNITY_EXCLUDE_STDDEF_H
251 ```

254 #### `UNITY_INCLUDE_PRINT_FORMATTED`
255 
256 Unity provides a simple (and very basic) printf-like string output implementation,
257 which is able to print a string modified by the following format string modifiers:
258 
259 - __%d__ - signed value (decimal)
260 - __%i__ - same as __%i__
261 - __%u__ - unsigned value (decimal)
262 - __%f__ - float/Double (if float support is activated)
263 - __%g__ - same as __%f__
264 - __%b__ - binary prefixed with "0b"
265 - __%x__ - hexadecimal (upper case) prefixed with "0x"
266 - __%X__ - same as __%x__
267 - __%p__ - pointer (same as __%x__ or __%X__)
268 - __%c__ - a single character
269 - __%s__ - a string (e.g. "string")
270 - __%%__ - The "%" symbol (escaped)
271 
272 _Example:_
273 ```C
274 #define UNITY_INCLUDE_PRINT_FORMATTED
275 
276 int a = 0xfab1;
277 UnityPrintFormatted("Decimal %d\n", -7);
278 UnityPrintFormatted("Unsigned %u\n", 987);
279 UnityPrintFormatted("Float %f\n", 3.1415926535897932384);
280 UnityPrintFormatted("Binary %b\n", 0xA);
281 UnityPrintFormatted("Hex %X\n", 0xFAB);
282 UnityPrintFormatted("Pointer %p\n", &a);
283 UnityPrintFormatted("Character %c\n", 'F');
284 UnityPrintFormatted("String %s\n", "My string");
285 UnityPrintFormatted("Percent %%\n");
286 UnityPrintFormatted("Color Red \033[41mFAIL\033[00m\n");
287 UnityPrintFormatted("\n");
288 UnityPrintFormatted("Multiple (%d) (%i) (%u) (%x)\n", -100, 0, 200, 0x12345);
289 ```

292 ### Toolset Customization
293 
294 In addition to the options listed above, there are a number of other options
295 which will come in handy to customize Unity's behavior for your specific
296 toolchain. It is possible that you may not need to touch any of these... but
297 certain platforms, particularly those running in simulators, may need to jump
298 through extra hoops to run properly. These macros will help in those
299 situations.
300 
301 
302 ##### `UNITY_OUTPUT_CHAR(a)`
303 
304 ##### `UNITY_OUTPUT_FLUSH()`
305 
306 ##### `UNITY_OUTPUT_START()`
307 
308 ##### `UNITY_OUTPUT_COMPLETE()`
309 
310 By default, Unity prints its results to `stdout` as it runs. This works
311 perfectly fine in most situations where you are using a native compiler for
312 testing. It works on some simulators as well so long as they have `stdout`
313 routed back to the command line. There are times, however, where the simulator
314 will lack support for dumping results or you will want to route results
315 elsewhere for other reasons. In these cases, you should define the
316 `UNITY_OUTPUT_CHAR` macro. This macro accepts a single character at a time (as
317 an `int`, since this is the parameter type of the standard C `putchar` function
318 most commonly used). You may replace this with whatever function call you like.
319 
320 _Example:_
321 Say you are forced to run your test suite on an embedded processor with no
322 `stdout` option. You decide to route your test result output to a custom serial
323 `RS232_putc()` function you wrote like thus:
324 ```C
325 #include "RS232_header.h"
326 ...
327 #define UNITY_OUTPUT_CHAR(a) RS232_putc(a)
328 #define UNITY_OUTPUT_START() RS232_config(115200,1,8,0)
329 #define UNITY_OUTPUT_FLUSH() RS232_flush()
330 #define UNITY_OUTPUT_COMPLETE() RS232_close()
331 ```
332 
333 _Note:_
334 `UNITY_OUTPUT_FLUSH()` can be set to the standard out flush function simply by
335 specifying `UNITY_USE_FLUSH_STDOUT`. No other defines are required.
336 
337 
338 ##### `UNITY_WEAK_ATTRIBUTE`
339 
340 ##### `UNITY_WEAK_PRAGMA`
341 
342 ##### `UNITY_NO_WEAK`
343 
344 For some targets, Unity can make the otherwise required setUp() and tearDown()
345 functions optional. This is a nice convenience for test writers since setUp and
346 tearDown don’t often actually do anything. If you’re using gcc or clang, this
347 option is automatically defined for you. Other compilers can also support this
348 behavior, if they support a C feature called weak functions. A weak function is
349 a function that is compiled into your executable unless a non-weak version of
350 the same function is defined elsewhere. If a non-weak version is found, the weak
351 version is ignored as if it never existed. If your compiler supports this feature,
352 you can let Unity know by defining UNITY_WEAK_ATTRIBUTE or UNITY_WEAK_PRAGMA as
353 the function attributes that would need to be applied to identify a function as
354 weak. If your compiler lacks support for weak functions, you will always need to
355 define setUp and tearDown functions (though they can be and often will be just
356 empty). You can also force Unity to NOT use weak functions by defining
357 UNITY_NO_WEAK. The most common options for this feature are:
358 
359 _Example:_
360 ```C
361 #define UNITY_WEAK_ATTRIBUTE weak
362 #define UNITY_WEAK_ATTRIBUTE __attribute__((weak))
363 #define UNITY_WEAK_PRAGMA
364 #define UNITY_NO_WEAK
365 ```
366

368 ##### `UNITY_PTR_ATTRIBUTE`
369 
370 Some compilers require a custom attribute to be assigned to pointers, like
371 `near` or `far`. In these cases, you can give Unity a safe default for these by
372 defining this option with the attribute you would like.
373 
374 _Example:_
375 ```C
376 #define UNITY_PTR_ATTRIBUTE __attribute__((far))
377 #define UNITY_PTR_ATTRIBUTE near
378 ```
379 
380 ##### `UNITY_PRINT_EOL`
381 
382 By default, Unity outputs \n at the end of each line of output. This is easy
383 to parse by the scripts, by Ceedling, etc, but it might not be ideal for YOUR
384 system. Feel free to override this and to make it whatever you wish.
385 
386 _Example:_
387 ```C
388 #define UNITY_PRINT_EOL { UNITY_OUTPUT_CHAR('\r'); UNITY_OUTPUT_CHAR('\n') }
389 ```

392 ##### `UNITY_EXCLUDE_DETAILS`
393 
394 This is an option for if you absolutely must squeeze every byte of memory out of
395 your system. Unity stores a set of internal scratchpads which are used to pass
396 extra detail information around. It's used by systems like CMock in order to
397 report which function or argument flagged an error. If you're not using CMock and
398 you're not using these details for other things, then you can exclude them.
399 
400 _Example:_
401 ```C
402 #define UNITY_EXCLUDE_DETAILS
403 ```
404

406 ##### `UNITY_EXCLUDE_SETJMP`
407 
408 If your embedded system doesn't support the standard library setjmp, you can
409 exclude Unity's reliance on this by using this define. This dropped dependence
410 comes at a price, though. You will be unable to use custom helper functions for
411 your tests, and you will be unable to use tools like CMock. Very likely, if your
412 compiler doesn't support setjmp, you wouldn't have had the memory space for those
413 things anyway, though... so this option exists for those situations.
414 
415 _Example:_
416 ```C
417 #define UNITY_EXCLUDE_SETJMP
418 ```

430 ## Getting Into The Guts
431 
432 There will be cases where the options above aren't quite going to get everything
433 perfect. They are likely sufficient for any situation where you are compiling
434 and executing your tests with a native toolchain (e.g. clang on Mac). These
435 options may even get you through the majority of cases encountered in working
436 with a target simulator run from your local command line. But especially if you
437 must run your test suite on your target hardware, your Unity configuration will
438 require special help. This special help will usually reside in one of two
439 places: the `main()` function or the `RUN_TEST` macro. Let's look at how these
440 work.
441 
442 
443 ##### `main()`
444 
445 Each test module is compiled and run on its own, separate from the other test
446 files in your project. Each test file, therefore, has a `main` function. This
447 `main` function will need to contain whatever code is necessary to initialize
448 your system to a workable state. This is particularly true for situations where
449 you must set up a memory map or initialize a communication channel for the
450 output of your test results.
451 
452 A simple main function looks something like this:
453 
454 ```C
455 int main(void) {
456  UNITY_BEGIN();
457  RUN_TEST(test_TheFirst);
458  RUN_TEST(test_TheSecond);
459  RUN_TEST(test_TheThird);
460  return UNITY_END();
461 }
462 ```
463 
464 You can see that our main function doesn't bother taking any arguments. For our
465 most barebones case, we'll never have arguments because we just run all the
466 tests each time. Instead, we start by calling `UNITY_BEGIN`. We run each test
467 (in whatever order we wish). Finally, we call `UNITY_END`, returning its return
468 value (which is the total number of failures).
469 
470 It should be easy to see that you can add code before any test cases are run or
471 after all the test cases have completed. This allows you to do any needed
472 system-wide setup or teardown that might be required for your special
473 circumstances.
474 
475 
476 ##### `RUN_TEST`
477 
478 The `RUN_TEST` macro is called with each test case function. Its job is to
479 perform whatever setup and teardown is necessary for executing a single test
480 case function. This includes catching failures, calling the test module's
481 `setUp()` and `tearDown()` functions, and calling `UnityConcludeTest()`. If
482 using CMock or test coverage, there will be additional stubs in use here. A
483 simple minimalist RUN_TEST macro looks something like this:
484 
485 ```C
486 #define RUN_TEST(testfunc) \
487  UNITY_NEW_TEST(#testfunc) \
488  if (TEST_PROTECT()) { \
489  setUp(); \
490  testfunc(); \
491  } \
492  if (TEST_PROTECT() && (!TEST_IS_IGNORED)) \
493  tearDown(); \
494  UnityConcludeTest();
495 ```
496 
497 So that's quite a macro, huh? It gives you a glimpse of what kind of stuff Unity
498 has to deal with for every single test case. For each test case, we declare that
499 it is a new test. Then we run `setUp` and our test function. These are run
500 within a `TEST_PROTECT` block, the function of which is to handle failures that
501 occur during the test. Then, assuming our test is still running and hasn't been
502 ignored, we run `tearDown`. No matter what, our last step is to conclude this
503 test before moving on to the next.
504 
505 Let's say you need to add a call to `fsync` to force all of your output data to
506 flush to a file after each test. You could easily insert this after your
507 `UnityConcludeTest` call. Maybe you want to write an xml tag before and after
508 each result set. Again, you could do this by adding lines to this macro. Updates
509 to this macro are for the occasions when you need an action before or after
510 every single test case throughout your entire suite of tests.
