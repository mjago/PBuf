1 # Adding Data to the Buffer
2 
3 ## Symbol meanings
4 For the sake of this document, I shall employ the following symbols in the examples that follow:
5 1. Priority is indicated with the letters `h`, `m`, and `l`, indicating, high, medium and low priorities. Note there
6 can be up to 8 levels of priority in practice, but here we will only deal with three for clarity. The number
7 following the letter indicates the number of elements of this priority in the buffer.
8 
9 2. A number following the priority letter indicates the order in which elements are *inserted into* the buffer, rather than the order they are *retrieved from* the buffer.
10 
11 2. `x` indicates unused but available storage in the buffer.
12 
13 3. `H` followed by a priority letter indicate a Head related to said priority, as in `Hm` representing *Head associated with the Medium priority buffer data.*
14 
15 4. `T` and `T*` represent *Tail* and *Tail pointer* respectively.
16 
17 ### Adding data to an empty buffer
18 
19 #### Initially:
20 
21 ```
22 x -> x -> x -> x -> ...
23 T*
24 ```
25 
26 Since the data to be added will be the only data in the buffer it will naturally be the highest priority
27 data on the buffer. In this case we can simply add the data at the element indicated by the tail pointer (T*)
28 and set the relevant head pointer (Hl* or Hm* or Hh*). We also need to set the active flag for the relevant
29 priority to indicate we have data of this priority in the buffer.
30 
31 ### Adding l1 becomes:
32 
33 ```
34 l1 -> x -> x -> x -> ...
35 T*
36 Hl*
37 ```

39 ## Adding data of the lowest priority to a partially full buffer containing only low priority
40 
41 ### Initially:
42 
43 ```
44 l1 -> l2 -> l3 -> x -> ...
45 T*
46  Hl Hl*
47 ```
48 
49 Since the buffer contains data in a high priority first, through lower priorities, to lower priority order,
50 to add data of the lowest priority we can simply add the data at the buffer position indicated by the relevant
51 head pointer H* and advance the Head Hl. The relevant active flag is already set, but we set it anyway since it
52 is quicker to just set the flag than test the flag and set it.
53 
54 ### Adding l4 becomes:
55 
56 ```
57 l1 -> l2 -> l3 -> l4 -> x ...
58 T*
59  Hl Hl*
60 ```

62 ## Adding data of the lowest priority to a partially full buffer containing mixed priorities
63 
64 ### Initially:
65 
66 ```
67 h1 -> h2 -> h3 -> m1 -> m2 -> m3 -> l1 -> x -> ...
68 T*
69  Hh Hm Hl Hl*
70 ```
71 
72 Similarly to the previous example, here we simply add the new element to the buffer at the position indicated
73 by the relevant head pointer. We then advance the relevant head and set the relevant active flag. Since the
74 added data is of the lowest priority it needs to be at the end of the buffer, so we need to do anything more
75 here.

85 ## Adding data of a high priority to a partially full buffer containing mixed priorities
86 
87 ### Initially:
88 
89 ```
90 h1 -> h2 -> h3 -> m1 -> m2 -> m3 -> l1 -> x -> ...
91 T*
92  Hh Hm Hl Hl*
93 ```
94 
95 We now wish to add high priority data to a buffer of mixed priorities. In this case we need the data to
96 be placed after data of a similar or higher priorities but before data of a lower priority. This is a two
97 stage process. Stage 1 involves where to store the data in the physical buffer and stage 2 involves
98 rearranging the buffer to set the data at the correct level of importance in the buffer, which is behind
99 than data of a higher or similar priority, but ahead of lower priority data.
100 
101 ### Adding h4 to the buffer:
102 
103 Since the buffer is not full we can add the data to an unused element (marked by x) that is pointed to by
104 the lowest active Head pointer:
105 
106 ### Stage 1:
107 
108 ```
109 h1 -> h2 -> h3 -> m1 -> m2 -> m3 -> l1 -> h4 -> x ...
110 T*
111  Hh Hm Hl Hl*
112 ```
113 
114 The buffer is now out of order since we have higher priority data that is behind low priority data. We
115 can correct this anomaly by remapping, or re-routing the buffer (since we have a link between adjacent
116 elements). This is achieved by reassigning links in such a way that the new entry is placed behind
117 data of higher and equal priority to itself, and ahead of data of lower priority to itself. This will
118 re-establish the correct prioritisation in the buffer.
119 
120 ### Finding the before insertion point:
121 
122 As indicated we need to place the data behind all data of higher or equal priority. To find this point
123 we simply need to examine the activity flags starting at the current priority, and find the first active
124 flag of equal or higher priority. If no such priority is active then we become the highest priority in
125 the buffer and need to be inserted at the front of the buffer. If higher or equal priority exists we
126 establish the priority closest to the current priority in increasing priority. In our case this is the
127 highest priority. Therefore we need to be placed between the Hh head pointer and the element following.
128 This is achieved by:
129 
130 1. saving the link at h3 in our case (which currently links to m1)
131 2. writing the position of the newly inserted data to the link at h3 (in our case the position of h4)
132 3. writing the current link of the newly inserted data to the pointer Hh* link (in our case m1)
133 4. writing the saved link to the current link (in our case h4*)
134 5. setting the relevant head pointer to indicate the new priority head.
135 6. setting the relevant priority activity flag incase it hasn't been set already.
136 
137 ### Stage 2:
138 
139 Below shows how the links are broken and rearranged as above. The affect is to place the element h4
140 behind the element h3 and ahead of m1.
141 
142 ```
143  ------------------------------>
144  <------------------------------------
145  ------->
146 h1 -> h2 -> h3 m1 -> m2 -> m3 -> l1 h4 x ...
147 T*
148 ```
149  Hm Hl Hh
150 
151 Below we see the buffer after redrawing and allowing for re-linking:
152 
153 ```
154 h1 -> h2 -> h3 -> h4 -> m1 -> m2 -> m3 -> l1 -> x ...
155 T*
156  Hh Hm Hl
157 ```

159 ## Adding data to a full buffer
160 
161 To add data to a full buffer we are required to push the oldest lowest priority data 'out the back' of
162 the buffer, since we no longer have room to accomodate additional data. This ensures that the limited
163 buffer space is reserved for the higher priority data and newer data of the lowest priority.
164 
165 ## Attempting to add data of a lower priority that currently exists in a full buffer
166 
167 If the data to be added is of a lower priority than any of the data in the buffer (indicated by an inactive
168 activity flag at the new priority), then the data is rejected, the buffer left untouched and an insertion
169 error returned to the caller.
170 
171 ## Adding data to a full buffer where the buffer is full of a single priority
172 
173 Where we are adding newer data to a full buffer containing only data of a similar priority we can simply overwrite
174 the oldest data in the buffer with our newest data, and advance the tail pointer to the next position. In this
175 way the latest insert is pushed to the back of the queue.
176

182  Hl

201 ## Adding data to a full buffer containing a mix of priority
202 
203 Assuming we have data of a higher or equal priority we can add it to the buffer by pushing less important
204 data (older or lower priority) out of the buffer. This is a two stage process - stage 1 is involved with
205 determining where to store the new data - stage 2 involves remapping or re-routing the links of the buffer
206 to insert the data between data of higher importance (higher priority or newer) and data of lower importance
207 (lower priority).
208 
209 ### Stage 1:
210 
211 #### To add m4:
212 
213 ```
214 h1 -> h2 -> h3 -> h4 -> m1 -> m2 -> m3 -> l1 -> l2
215 T* T
216  Hh Hm Hl
217 ```
218 
219 Here we have a full buffer containing a mix of prioritisation. To add our data (m4) we need to determine
220 the element containing the lowest priority and oldest element. In our case this is the element l1. Regardless
221 of which priority we are adding this is the element to get overwritten since it it has the least importance.
222 
223 ### Overwriting:
224 
225 ```
226 h1 -> h2 -> h3 -> h4 -> m1 -> m2 -> m3 -> m4 -> l2
227 T* T
228  Hh Hm Hl
229 ```
230 
231 Here we need to overwrite l1 with m4 as shown. Since we haven't a direct pointer to l1 we need to find
232 l1 some other way. l1 can be considered the 'tail' of priority 'l'. This can be determined by finding the
233 next highest priority on the buffer (that is the next active priority) and following the link of that
234 priority's relevant head. In our case the next highest priority to l1 is priority m. We follow the link
235 given by the head Hm to l1 and overwrite m4 there.
236 
237 In this case the buffer is still in the correct order and we simply need to advance the Hm head:
238 
239 ```
240 h1 -> h2 -> h3 -> h4 -> m1 -> m2 -> m3 -> m4 -> l2
241 T* T
242  Hh Hm Hl
243 ```
244

291 ## Retrieving Data from the buffer:
292 
293 ### Initially:
294 
295 ```
296 l1 -> l2 -> l3 -> l4 -> l5 -> l6 -> l7 -> l8 -> l9 -> l10
297 T* T
298  Hl
299 ```
300 
301 To remove data from the buffer we first check that the buffer is not empty. If it is empty an
302 error is returned to the caller. If the buffer is not empty we return the element pointed to
303 by the Tail pointer T* and advance the Tail pointer.
304

316 priority and need to make the priority inactive:
